# Internationalisation

Supported locales: **English (en)**, **Swedish (sv)**, **German (de)**,
**Spanish (es)**, **French (fr)**, **Portuguese (pt)**.

Both apps use i18next with `fallbackLng: 'en'`, so any key without a
translation renders the English string rather than a raw key.

## Where things live

| | Web | Mobile |
|---|---|---|
| Config | `frontend/src/i18n/config.ts` | `mobile/src/i18n/index.ts` |
| Locale files | `frontend/src/i18n/locales/*.json` | `mobile/src/i18n/locales/*.json` |
| Translation source | `frontend/scripts/web-translations.json` | `mobile/scripts/mobile-translations.json` |
| Generator | `frontend/scripts/gen-locales.cjs` | `mobile/scripts/gen-locales.cjs` |

`en.json` is the **source of truth** in both apps: it defines the key set and
the key order. The other locale files are generated — do not hand-edit them.

The web namespace covers the public site (`common`, `header`, `landing`,
`auth`, `about`, `careers`, `partners`, `contact`, `support`, `blog`,
`notFound`, `footer`) and the whole advertiser dashboard (`dashboard.*`).
Mobile keeps everything under `mobile.*`.

## Adding or changing a string

1. Add the key to `en.json` (web or mobile).
2. Use it: `const { t } = useTranslation()` then `t('some.key')`.
3. Add translations to the matching `*-translations.json`.
4. Regenerate:

   ```sh
   node frontend/scripts/gen-locales.cjs
   node mobile/scripts/gen-locales.cjs
   ```

The generator prints `complete` or lists every untranslated key, and exits
non-zero while any gap remains — so it doubles as the coverage report.

It also **fails hard** if a translation drops or invents an interpolation
placeholder (`{{count}}`, `{{email}}`, …), because that breaks the string at
runtime rather than merely reading oddly.

## Adding a language

1. Add the code to `LANGS` in both generators.
2. Add its translations to both `*-translations.json`.
3. Register it in `languages` and `resources` in both i18n configs.

The web language switcher, the mobile language picker and the dashboard
Settings language dropdown all read `languages`, so they pick it up
automatically — there is no fourth list to remember.

## Conventions

- Config arrays hold **keys, not copy**. Icons, gradients, numbers and
  initials stay in the component; the text lives in the locale files and is
  looked up by key.
- Zod schemas that carry validation messages are built **inside** the
  component (`buildSchema(t)`), so the messages follow a language change
  instead of being frozen at module load.
- Language names stay in their own language (Deutsch, Svenska) — never
  translated.
- Job titles and departments on the careers page stay in English on purpose:
  they are the names the roles are advertised under.

## Current coverage

Run the generators to see the live numbers. Both apps are currently complete
in all six languages:

- **Mobile** — 331 keys × 6 locales.
- **Web** — 963 keys × 6 locales.

A handful of strings are deliberately identical to English in some locales —
the brand name, product names (Ad Journey Storyteller), metric acronyms
(CTR, ROAS, CPC) and loanwords German and French use natively (Dashboard,
Budget, Feedback). The parity check ignores these; they are not gaps.

The non-English copy has **not yet had a native-speaker review**. Treat it as
a complete first pass, not as final launch copy — see
[TRANSLATION_REVIEW.md](TRANSLATION_REVIEW.md) for the reviewer handoff,
the domain glossary to agree with the client, and the per-locale sign-off
table.
